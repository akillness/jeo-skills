### Fixed
- **`llm-wiki` — jeo install (Step 3e) diverged from the skill's own vault schema**. `setup-all-skills-prompt.md` Step 3e bootstrapped `~/vaults/llm-wiki` with an ad hoc `mkdir -p raw wiki && touch index.md log.md`, instead of calling `.agent-skills/llm-wiki/scripts/bootstrap-vault.sh` — the same script Step 6's Knowledge Pipeline block already reused correctly. A fresh `jeo` install therefore produced a vault missing `raw/sources`, `raw/assets`, `wiki/{sources,entities,concepts,queries,reports}`, the `AGENTS.md` schema contract, and the templated `index.md`/`log.md`. Step 3e now calls `bootstrap-vault.sh "$WIKI_VAULT"` (falling back to the old minimal skeleton only if the skill isn't installed yet), so the install-time vault and the skill's documented schema never drift apart. Verified in a scratch `$HOME` sandbox: full schema created on first run, idempotent no-op on re-run, and the minimal fallback still works when the skill directory is absent. Also documented the `$LLM_WIKI_VAULT` convention (default `~/vaults/llm-wiki`) directly in `SKILL.md` Step 1, so manual `bootstrap-vault.sh` runs target the same fixed global vault `jeo`'s tool-flow rule and post-turn hook already expect.

- **`graphify` — durable wikilink normalization patch** (`2.0.0` → `2.1.0`). graphify (the PyPI `graphifyy` package) saves community / god-node wiki pages under a *slugged* filename (`Community 36` → `Community_36.md`) but its generator emits *raw-label* wikilinks (`[[Community 36]]`) that never resolve, producing broken links across the whole vault (691 in one vault). This is a generator-side bug, so regex-fixing the generated files does not survive the next rebuild. Added `.agent-skills/graphify/scripts/patch_wikilink.py` — an idempotent, self-testing patcher that rewrites every raw-label link site in `site-packages/graphify/wiki.py` to `[[slug|label]]` via a `_wikilink` helper. Because the in-place edit is wiped by `pip install --upgrade graphifyy`, the skill now documents wiring the patcher into config so it self-heals (jeo: the `post-implementation` hook ahead of `graphify update .`). New `wikilink-normalization-patch` recipe + failure-pattern row in `references/build-and-fallback-recipes.md`, best-practice + reference in `SKILL.md`, and an extra rule in `SKILL.toon`. Permanent fix tracked as an upstream PR to `safishamsi/graphify`.
- **`deep-research` — coverage-gate false-pass + verbatim validator path** (`1.0.0` → `1.0.1`). `scripts/validate_json.py` exited `0` when a JSON path passed via `-j` did not exist, so the deep phase (which treats a clean exit as "item done") could mark a never-written item complete. The validator now counts missing requested files and exits non-zero when any are absent or nothing validated, while an empty `--dir` scan still exits `0`. Also fixed the deep-pipeline prompt template, which hardcoded the upstream `~/.claude/skills/research/validate_json.py` path that does not exist in a jeo-skills install: introduced a `{validate_script}` substitution variable (kept verbatim-substitutable) resolving to the shipped `.agent-skills/deep-research/scripts/validate_json.py`, and updated the one-shot example + Validation Gate notes to match.
- **`deep-research` — dynamic validator-path resolution** (`1.0.1` → `1.0.2`). The `1.0.1` fix still resolved `{validate_script}` to a single hardcoded literal (`.agent-skills/deep-research/scripts/validate_json.py`), and the one-shot example / Validation Gate carried two *other* hardcoded literals — so the prompt broke whenever the skill ran from a different install root. `{validate_script}` is now resolved **dynamically from the live filesystem**: a Step-3 `ls … | head -n1` probe picks the first validator that actually exists across the real install roots (`~/.agents/skills/…`, the repo checkout `$PWD/.agent-skills/…`, `~/.claude/skills/deep-research/…`, upstream `~/.claude/skills/research/…`) and aborts with a clear error when none is found. The two prompt-text call sites (prompt template, one-shot example) substitute `{validate_script}` to the resolved absolute path before the prompt is handed to a subagent, while the orchestrator-run Validation Gate uses `$validate_script` directly. Verified: the probe resolves to the repo copy from the checkout and to `~/.agents/skills/…` from an unrelated cwd, and the gate still passes valid JSON (exit 0) / fails a missing `-j` file (exit 1) through the resolved path.
- **`deep-research` — ponytail de-dup + one-shot call-site fix** (`1.0.2` → `1.0.3`). The `1.0.2` one-shot example rendered `python "$validate_script"`, but that example is prompt text handed to a spawned web-search subagent whose shell has no `$validate_script` — so the rendered validation command would have failed. Switched it to the brace placeholder `python {validate_script}` (substituted to an absolute path before send), matching the prompt template; only the orchestrator-run Validation Gate keeps the genuine `$validate_script` shell var. Also collapsed the three-way explanation of dynamic resolution to a single source of truth — the Step-3 probe plus the one-line pointer comment — deleting the duplicate justification paragraph.
