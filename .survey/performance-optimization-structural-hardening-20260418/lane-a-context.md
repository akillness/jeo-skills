## Workflow Context
- **CLI / dev workflow:** measurement-led tuning shows up in shell startup, local rebuild/reload speed, benchmark loops, and CI runtime. Chrome DevTools explicitly frames performance work as **recording runtime performance** to find bottlenecks; for command-line workflows, adjacent tooling like [hyperfine](https://github.com/sharkdp/hyperfine) exists solely as a **command-line benchmarking tool**. In CI/web automation, [Lighthouse CI](https://github.com/GoogleChrome/lighthouse-ci) is used to **run Lighthouse for every commit, view changes, and prevent regressions**.
- **Web frontend / backend / fullstack:** measurement-first workflows are standard across browser traces, Lighthouse audits, APM traces, and DB plan analysis. Chrome documents DevTools Performance as the place to **evaluate runtime performance** and record traces ([Chrome DevTools Performance](https://developer.chrome.com/docs/devtools/performance)); Lighthouse is an **automated tool** with audits for **performance, accessibility, SEO, and more** ([Lighthouse overview](https://developer.chrome.com/docs/lighthouse/overview)); PostgreSQL says plan choice is **absolutely critical for good performance** and directs users to inspect plans with [`EXPLAIN`](https://www.postgresql.org/docs/current/using-explain.html).
- **Product / UX / ops:** measurement-led tuning appears where teams need field-data dashboards, SLO-ish regressions, and before/after validation rather than one-off tips. A WordPress Performance proof-of-concept explicitly centers collecting **Core Web Vitals field data** so site owners can **see what real users are experiencing** inside the dashboard ([WordPress Performance Dashboard PR](https://github.com/WordPress/performance/pull/1098)).
- **Marketing / content / automation:** performance work shows up in SEO/CWV review, landing-page audits, CMS/plugin optimization, and regression gates in CI. Lighthouse’s inclusion of **SEO** alongside performance makes it a shared artifact across engineering and growth/content teams ([Lighthouse overview](https://developer.chrome.com/docs/lighthouse/overview)); WordPress Performance discussions show site owners asking how to measure improvement in **Core Web Vitals** for speculative loading, not just whether a feature “works” ([WordPress issue #1426](https://github.com/WordPress/performance/issues/1426)).
- **Game development:** measurement-led tuning is explicitly profiler-led. Unity documents the [Unity Profiler](https://docs.unity3d.com/Manual/Profiler.html) as a tool to get **performance information** about an application on target devices and inspect resource allocation during development; Epic’s [Unreal Insights](https://dev.epicgames.com/documentation/en-us/unreal-engine/unreal-insights-in-unreal-engine) is described as the way to **profile your project's performance**.

## Affected Users
- **Individual developers:** shell/plugin users, frontend engineers, API engineers, DB-heavy fullstack developers, and game programmers all hit slow feedback loops where the first question is “what is actually slow?” rather than “what should I rewrite?” Sources: [hyperfine](https://github.com/sharkdp/hyperfine), [Chrome DevTools Performance](https://developer.chrome.com/docs/devtools/performance), [PostgreSQL EXPLAIN](https://www.postgresql.org/docs/current/using-explain.html), [Unity Profiler](https://docs.unity3d.com/Manual/Profiler.html), [Unreal Insights](https://dev.epicgames.com/documentation/en-us/unreal-engine/unreal-insights-in-unreal-engine).
- **Teams running CI / release gates:** they need repeatable measurement artifacts to catch regressions, especially for web apps and automation-heavy repos. Source: [Lighthouse CI](https://github.com/GoogleChrome/lighthouse-ci), [actions/runner #1031](https://github.com/actions/runner/issues/1031).
- **Site owners / product / growth / content stakeholders:** they care about field CWV, SEO, and real-user responsiveness but often consume evidence through dashboards/plugins rather than profilers. Source: [WordPress Performance Dashboard PR](https://github.com/WordPress/performance/pull/1098), [Lighthouse overview](https://developer.chrome.com/docs/lighthouse/overview).
- **Game teams and technical artists:** they need frame-time, memory, and resource-allocation evidence tied to target hardware before changing assets, effects, or engine code. Source: [Unity Profiler](https://docs.unity3d.com/Manual/Profiler.html), [Unreal Insights](https://dev.epicgames.com/documentation/en-us/unreal-engine/unreal-insights-in-unreal-engine).

## Current Workarounds
- **Manual profiling hops:** users jump among browser Network/Performance panels, Lighthouse/WebPageTest, APM traces, slow-query logs, and `EXPLAIN ANALYZE`, then manually reconcile findings. Sources: [Chrome DevTools Performance](https://developer.chrome.com/docs/devtools/performance), [Lighthouse overview](https://developer.chrome.com/docs/lighthouse/overview), [PostgreSQL EXPLAIN](https://www.postgresql.org/docs/current/using-explain.html).
- **Local benchmark wrappers and ad hoc scripts:** command-line benchmarking tools like [hyperfine](https://github.com/sharkdp/hyperfine) or one-off timing scripts compensate when a workflow has no built-in measurement loop.
- **CI wrappers for regression checking:** teams bolt on [Lighthouse CI](https://github.com/GoogleChrome/lighthouse-ci) to make perf regressions visible per commit because default pipelines usually do not provide strong performance diffing.
- **Dashboard/plugin stopgaps:** WordPress Performance experiments collect field CWV or speculative-loading measurements inside admin UIs because many site owners will not open DevTools or profiler traces directly ([PR #1098](https://github.com/WordPress/performance/pull/1098), [issue #1426](https://github.com/WordPress/performance/issues/1426)).
- **Engine-specific profiler capture interpretation:** in games, teams already split work between general optimization guidance and engine-specific profiler reading via Unity/Unreal tooling ([Unity Profiler](https://docs.unity3d.com/Manual/Profiler.html), [Unreal Insights](https://dev.epicgames.com/documentation/en-us/unreal-engine/unreal-insights-in-unreal-engine)).

## Adjacent Problems
- **Observability vs tuning:** many workflows first need telemetry rollout, dashboards, or field data collection before tuning can begin; WordPress’s CWV dashboard work is a good example ([PR #1098](https://github.com/WordPress/performance/pull/1098)).
- **Debugging vs optimization:** complaints often start as “it feels slow” or “reload takes forever,” but the underlying problem may be a bug, excessive logging, or incorrect plugin behavior rather than a classic hot path. Sources: [Next.js discussion #17977](https://github.com/vercel/next.js/discussions/17977), [oh-my-zsh issue #13505](https://github.com/ohmyzsh/ohmyzsh/issues/13505), [actions/runner #1031](https://github.com/actions/runner/issues/1031).
- **Structural/code-quality work vs bounded tuning:** once evidence identifies the hotspot, the next step often becomes refactoring, architecture cleanup, indexing/schema redesign, or asset-pipeline changes rather than more profiling. Sources: [PostgreSQL EXPLAIN](https://www.postgresql.org/docs/current/using-explain.html), [Unity Profiler](https://docs.unity3d.com/Manual/Profiler.html).
- **Environment mismatch:** local dev, CI, staging, production, and target hardware can show different bottlenecks, which is why both Unity and WordPress emphasize target-device / real-user measurement ([Unity Profiler](https://docs.unity3d.com/Manual/Profiler.html), [PR #1098](https://github.com/WordPress/performance/pull/1098)).

## User Voices

- **CI / automation:** “A step that takes **less than a second** to actually execute can easily take **10s of minutes or even hours** to be considered complete…” — user report in [actions/runner issue #1031](https://github.com/actions/runner/issues/1031)
- **Marketing / content / site-owner workflow:** “**What is the best way to measure improvement in Core Web Vitals?** A Lighthouse Report and a WebPageTest will not trigger the user intent.” — user report in [WordPress Performance issue #1426](https://github.com/WordPress/performance/issues/1426)
