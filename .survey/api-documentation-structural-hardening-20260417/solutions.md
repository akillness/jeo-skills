# Solution Landscape: api-documentation structural hardening

## Solution List
| Name | Approach | Strengths | Weaknesses | Notes |
|------|----------|-----------|------------|-------|
| Redocly / Redoc | Spec-driven API reference plus docs platform | Strong reference UX, governance tooling, docs-as-code workflow, external + internal surfaces | Needs editorial structure beyond generated reference; large surface can still sprawl | Frequently appears in API-doc tooling roundups |
| ReadMe | Hosted developer portal with guides, recipes, changelog, explorer | Good onboarding and narrative guide support | Simpler hosted workflow can be less Git/CI-native; structure still needs manual curation | Strong for portal-first teams |
| Bump.sh | OpenAPI/AsyncAPI portal with changelog and diffing | Good for published reference + change management | Generation does not replace quickstarts or workflow guides | Useful for anti-drift patterns |
| Scalar | Interactive API docs and API client | Strong DX and clean reference UI | Less comprehensive governance/customization than heavier suites | Often paired with other docs systems |
| Stoplight | Design/governance suite with interactive docs | Strong design-first ecosystem, reusable docs renderer | Navigation and grouping issues recur on large APIs | More lifecycle platform than docs-only tool |
| Mintlify / GitBook / Docusaurus | Docs platform / docs-as-code shells | Flexible for mixed narrative + reference docs | Need deliberate structure; can blur API docs with broader product docs | Useful when many non-reference pages are needed |
| Fern / Speakeasy / SDK-linked stacks | Spec + SDK + docs generation | Reduce some drift between API, SDK, and docs | Still need hand-written guides, migration notes, and trustworthy examples | Good evidence that docs are multi-surface |
| Stripe / GitHub docs patterns | Primary-source exemplars rather than products | Show durable split between reference, quickstart, troubleshooting, changelog, migration, webhook guides | Not turnkey tooling | Best shape reference for this skill rewrite |

## Categories
- **Developer portals:** ReadMe, Mintlify, Redocly, Bump.sh, GitBook
- **Interactive reference renderers:** Redoc, Scalar, Swagger UI, Stoplight Elements
- **Docs-as-code / hybrid docs stacks:** Docusaurus, Mintlify, GitBook, Redocly Revel
- **Spec-first governance suites:** Stoplight, SwaggerHub, Redocly
- **SDK-linked docs stacks:** Fern, Speakeasy, related generators
- **Drift / changelog tooling:** Bump.sh, oasdiff, API Diff, Redocly CLI
- **Operational webhook / event guidance:** AsyncAPI ecosystem, vendor webhook docs such as Stripe and Svix

## What People Actually Use
Teams usually pair **generated or rendered reference** with **hand-written quickstarts, guides, SDK docs, webhook pages, troubleshooting, changelog, and migration notes**. Mature portals such as Stripe, GitHub, ReadMe, and Redocly all separate these surfaces instead of pretending the spec alone is enough.

## Frequency Ranking

## Key Gaps
- Generated reference does not solve **mode selection** for quickstart vs reference vs SDK vs webhook vs migration work.
- Large API surfaces still need deliberate **navigation and grouping**, not just more autogenerated pages.
- “Always in sync” tooling does not remove the need for **editorial review, truthful examples, and anti-drift checks**.
- Multi-surface docs systems still need a clear **front-door router** so maintainers do not collapse everything into one giant page.

## Contradictions
- Vendors market API docs as generated, synced, and easy, but user issues and vendor docs both show that teams still hand-curate examples, sidebars, grouping, recipes, changelogs, and migration notes.
- Tooling often claims a single source of truth, while real portals split content across multiple source layers: contract spec, portal pages, SDK docs, webhook/event pages, and changelog/update surfaces.

## Key Insight
The repo does not need another API-doc skill. It needs `api-documentation` to behave more like the best mature API-doc surfaces: a small routing front door that classifies the job quickly, then hands detailed mode templates, example quality checks, and anti-drift guidance to focused support documents.
