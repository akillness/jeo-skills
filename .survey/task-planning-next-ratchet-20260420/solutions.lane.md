# Lane B — Solutions

## Solutions
| Name | Approach | Strengths | Weaknesses | Notes |
|---|---|---|---|---|
| Atlassian planning guidance | Backlog refinement + sprint planning | Strong readiness and sprint framing | Can drift into process/process-overhead language | Useful planning canon |
| GitHub Projects + milestones | Repo-native planning/tracking surfaces | Close to issues/PRs and execution state | Weak alone for compact packet authoring | System of record, not packet front door |
| HacknPlan | Game-aware milestone planning | Best game-specific structure in this run | Still a tracking tool, not a packet skill | Useful game reference |
| Productboard-style roadmap tooling | Strategy-to-delivery handoff | Strong for roadmap intent and prioritization | Needs downstream execution slicing elsewhere | Contrast case for roadmap-to-delivery translation |
| Docs / spreadsheets / Notion | Manual planning artifacts | Flexible, fast, low-friction | Drift, duplicate truth, weak handoffs | Common workaround pattern |

## Frequency Ranking
1. Board-centric planning/tracking tools.
2. Manual docs/spreadsheets/Notion sidecars.
3. Roadmap surfaces that still require downstream decomposition.

## Categories
- Systems of record.
- Strategy/roadmap translation tools.
- Manual planning sidecars.

## Early Takeaway
The landscape is crowded with places to store work, but much thinner on compact, portable planning packets that explicitly separate readiness, blockers, sequencing, and route-outs across developer workflow, web/fullstack, PM/ops, marketing/GTM, and game work.
