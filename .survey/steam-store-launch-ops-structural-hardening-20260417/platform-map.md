# Platform Map: steam-store-launch-ops structural hardening

## Settings
| Concern | Current team knobs | Common Layer |
|---------|--------------------|--------------|
| Page promise | capsule, screenshots, trailer hook, short description, tags | promise clarity |
| Visibility | announcement timing, creator beats, event participation, demo exposure | visibility acquisition |
| Proof | demo existence, demo quality, first-session flow, festival build polish | proof / readiness |
| Timing | Next Fest registration, review lead time, launch window, freeze dates | timing / hook fit |
| Ops | asset completeness, creator packet, checklist ownership, launch sequencing | ops readiness |

## Rules
| Concern | Practical heuristic | Common Layer |
|---------|---------------------|--------------|
| Weak wishlists | Split traffic problems from conversion problems before changing assets | diagnosis first |
| Demo question | If buyer uncertainty is the real issue, the best intervention may be demo proof instead of more page copy | proof over copy when needed |
| Event prep | Treat Next Fest as a live readiness gate, not just a promo checkbox | event readiness |
| Launch confidence | Do not recommend hard launch commitments without review/process slack | operational guardrail |
| Asset critique | Prefer the shortest path to validated learning rather than polishing every asset at once | one experiment first |

## Hooks
| Lifecycle | Examples | Common Layer |
|-----------|----------|--------------|
| Coming Soon publish | first page launch, first capsule/trailer/tag pass | page baseline hook |
| Wishlist anomaly | post-announcement dip/spike, social traffic with weak conversion | diagnosis hook |
| Demo decision | ship/update/hide demo, choose festival build | proof hook |
| Event window | Next Fest registration, page lock, livestream prep, event-week staffing | event hook |
| Launch gate | review submission, checklist completion, release approval, launch-day ops | release hook |

## Platform Gaps
- The current skill needs a stronger normalized split between `visibility`, `promise`, `proof`, `timing`, and `ops` instead of treating most requests as page audits with extra checklists.
- Demo / Next Fest work should act like a distinct readiness branch, not an add-on section.
- The output packet should make the intervention type explicit: what bottleneck, why now, what evidence, and what next artifact.
- Steam’s official docs are function-by-function; the skill should translate them into one chronological launch workflow.
