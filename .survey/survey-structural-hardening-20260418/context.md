# Context: survey structural hardening

## Workflow Context
The current `survey` skill is both oversized and highly connected: `graphify-out/GRAPH_REPORT.md` lists it at **392 lines** and **degree 17**, and the same report says the repo's next wins come from **dense-anchor cleanup** rather than more wrappers ([`../../graphify-out/GRAPH_REPORT.md`](../../graphify-out/GRAPH_REPORT.md)). That creates front-door friction because `survey` is the reusable discovery entry point before planning or implementation, and it is expected to write `.survey/{slug}/` artifacts that later skills consume ([`../../.agent-skills/survey/SKILL.md`](../../.agent-skills/survey/SKILL.md)).

That friction shows up across the repo's main maintenance lanes because those lanes have already been hardened into narrower routing-first anchors:
- **CLI / developer workflow:** the developer-workflow cluster now separates `file-organization`, `workflow-automation`, `git-workflow`, `git-submodule`, `npm-git-install`, `fabric`, and `lmstudio-cli`, and explicitly says the higher-value move is shrinking dense anchors instead of adding overlapping wrappers ([`../../.agent-skills-wiki/concepts/developer-workflow-cluster.md`](../../.agent-skills-wiki/concepts/developer-workflow-cluster.md)).
- **Web / fullstack:** frontend maintenance depends on compact metadata and side-by-side discovery surfaces, so duplicated or over-broad front doors create activation noise; the frontend concept page says the best fix is canonicalization plus sharper trigger wording, not another wrapper ([`../../.agent-skills-wiki/concepts/frontend-skill-canonicalization.md`](../../.agent-skills-wiki/concepts/frontend-skill-canonicalization.md)).
- **PM / ops:** `task-planning` was just tightened into a routing-first anchor that separates discovery from delivery across developer, web/fullstack, product/ops, marketing, and game work, which increases the cost of leaving `survey` as a large mixed front door near that boundary ([`../../README.md`](../../README.md)).
- **Marketing / content:** the marketing cluster is now "broad intake first, specialist lane second," while the documentation cluster splits internal docs, API docs, user guides, changelogs, and presentations into separate anchors; an overgrown `survey` front door adds another broad intake layer in front of already-sharpened lanes ([`../../.agent-skills-wiki/concepts/marketing-cluster.md`](../../.agent-skills-wiki/concepts/marketing-cluster.md), [`../../.agent-skills-wiki/concepts/documentation-cluster.md`](../../.agent-skills-wiki/concepts/documentation-cluster.md)).
- **Game-development maintenance:** the game cluster now follows "coordination first, specialist depth second" and depends on explicit discovery of narrow skills like `game-build-log-triage`, `game-performance-profiler`, and `steam-store-launch-ops`; a large generic survey front door makes that specialist routing less direct ([`../../.agent-skills-wiki/concepts/game-development-cluster.md`](../../.agent-skills-wiki/concepts/game-development-cluster.md)).

## Current Workarounds
1. **Use the support packet instead of rereading the whole front door.** The repo already added `survey/references/evidence-recovery-ladder.md`, `survey/evals/evals.json`, and `SKILL.toon` so maintainers can rely on the smaller support surfaces for recovery rules and trigger memory instead of treating `SKILL.md` as the only source of truth ([`../../.agent-skills/survey/references/evidence-recovery-ladder.md`](../../.agent-skills/survey/references/evidence-recovery-ladder.md), [`../../.agent-skills/survey/evals/evals.json`](../../.agent-skills/survey/evals/evals.json), [`../../.agent-skills/survey/SKILL.toon`](../../.agent-skills/survey/SKILL.toon)).
2. **Lean on prior wiki/query memory to remember what `survey` is for.** The repo-local query note defines `survey` as the evidence-recovery gate and records that the bounded fix was hardening this main entry point, not adding a second nearby research skill ([`../../.agent-skills-wiki/queries/survey-evidence-recovery-2026-04-12.md`](../../.agent-skills-wiki/queries/survey-evidence-recovery-2026-04-12.md)).
3. **Route quickly into domain anchors once the survey output exists.** Multiple cluster notes say the next structural win is boundary clarity and smaller front doors, so maintainers currently avoid keeping domain detail inside `survey` and instead hand off to the sharper anchors in developer workflow, frontend, PM, documentation, marketing, and game lanes ([`../../.agent-skills-wiki/concepts/developer-workflow-cluster.md`](../../.agent-skills-wiki/concepts/developer-workflow-cluster.md), [`../../.agent-skills-wiki/concepts/frontend-skill-canonicalization.md`](../../.agent-skills-wiki/concepts/frontend-skill-canonicalization.md), [`../../.agent-skills-wiki/concepts/documentation-cluster.md`](../../.agent-skills-wiki/concepts/documentation-cluster.md), [`../../.agent-skills-wiki/concepts/game-development-cluster.md`](../../.agent-skills-wiki/concepts/game-development-cluster.md)).
4. **Depend on discovery-surface sync outside the skill itself.** The graph report and support-coverage note both emphasize README / setup / manifest / compact surfaces as high-maintenance discovery nodes, so maintainers currently use those external surfaces to keep `survey` discoverable and truthful even though the front door remains large ([`../../graphify-out/GRAPH_REPORT.md`](../../graphify-out/GRAPH_REPORT.md), [`../../.agent-skills-wiki/concepts/skill-support-coverage.md`](../../.agent-skills-wiki/concepts/skill-support-coverage.md)).

## Adjacent Problems
- The graph report says discovery docs remain the highest-degree maintenance nodes, so any oversized front-door skill adds review and sync pressure across README, setup, manifest, and compact surfaces, not just inside one folder ([`../../graphify-out/GRAPH_REPORT.md`](../../graphify-out/GRAPH_REPORT.md)).
- The support-coverage note says coverage is already complete and that the next wins come from trigger, boundary, and maintenance-shape improvements, which makes `survey`'s size a structural quality problem rather than a missing-support-files problem ([`../../.agent-skills-wiki/concepts/skill-support-coverage.md`](../../.agent-skills-wiki/concepts/skill-support-coverage.md)).
- `survey` sits close to other high-visibility routers (`task-planning`, `jeo`, `ralph`) that have already been hardened by moving slower-changing detail into references; leaving `survey` oversized makes that maintenance pattern inconsistent in the repo's research/planning layer ([`../../README.md`](../../README.md), [`../../.agent-skills-wiki/concepts/skill-support-coverage.md`](../../.agent-skills-wiki/concepts/skill-support-coverage.md)).
- Documentation, frontend, and game clusters all explicitly warn against umbrella skills swallowing narrower jobs; an oversized `survey` front door increases the chance that early discovery blurs those boundaries before routing happens ([`../../.agent-skills-wiki/concepts/documentation-cluster.md`](../../.agent-skills-wiki/concepts/documentation-cluster.md), [`../../.agent-skills-wiki/concepts/frontend-skill-canonicalization.md`](../../.agent-skills-wiki/concepts/frontend-skill-canonicalization.md), [`../../.agent-skills-wiki/concepts/game-development-cluster.md`](../../.agent-skills-wiki/concepts/game-development-cluster.md)).
