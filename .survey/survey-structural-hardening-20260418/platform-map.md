# Platform Map: survey structural hardening

## Settings
| Concern | Claude | Codex | Gemini | Common Layer |
|---------|--------|-------|--------|--------------|
| Config location and precedence | Claude uses scoped JSON settings across managed, user, project, and local project files, with project scope intended for team-shared hooks, permissions, and MCP servers. ([Claude settings](https://code.claude.com/docs/en/settings)) | Codex uses `~/.codex/config.toml` plus trusted-project `.codex/config.toml`, with CLI/profile/project/user/system precedence. ([Codex config basics](https://developers.openai.com/codex/config-basic)) | Gemini layers defaults, system files, user `~/.gemini/settings.json`, project `.gemini/settings.json`, env vars, and CLI args. ([Gemini configuration](https://raw.githubusercontent.com/google-gemini/gemini-cli/main/docs/reference/configuration.md)) | Keep `survey` behavior in checked-in repo files, not only in platform-native config. |
| Safety defaults | Claude supports permission controls and permission hooks. ([Claude hooks](https://code.claude.com/docs/en/hooks)) | Codex documents sandbox and approval behavior in config/security docs. ([Codex config reference](https://developers.openai.com/codex/config-reference), [Codex security](https://developers.openai.com/codex/security)) | Gemini exposes approval modes including a read-only plan mode. ([Gemini configuration](https://raw.githubusercontent.com/google-gemini/gemini-cli/main/docs/reference/configuration.md)) | Assume least privilege and one deterministic artifact write. |
| MCP/external tools | Claude project settings can share MCP servers. ([Claude settings](https://code.claude.com/docs/en/settings)) | Codex config documents MCP servers and optional parallel tool calls. ([Codex config reference](https://developers.openai.com/codex/config-reference)) | Gemini treats MCP tools and hook matchers as first-class runtime surfaces. ([Gemini configuration](https://raw.githubusercontent.com/google-gemini/gemini-cli/main/docs/reference/configuration.md), [Gemini hooks](https://raw.githubusercontent.com/google-gemini/gemini-cli/main/docs/hooks/reference.md)) | MCP is optional; the survey artifact schema must not depend on it. |

## Rules
| Concern | Claude / OMC | Codex / OMX | Gemini / OHMG | Common Layer |
|---------|---------------|-------------|---------------|--------------|
| Research-worker abstraction | Claude has first-class subagents with custom prompts, tool access, and permissions. ([Claude sub-agents](https://code.claude.com/docs/en/sub-agents)) | The supplied Codex docs emphasize instruction files, approvals, and config rather than a matching subagent model. ([Codex config reference](https://developers.openai.com/codex/config-reference)) | Gemini has first-class subagents with focused context and tool restrictions. ([Gemini subagents](https://raw.githubusercontent.com/google-gemini/gemini-cli/main/docs/core/subagents.md)) | The 4-lane survey structure matters, but the survey instructions must remain complete without subagent-only features. |
| Durable instruction surface | Claude can use project/user assets and subagent markdown. ([Claude settings](https://code.claude.com/docs/en/settings), [Claude sub-agents](https://code.claude.com/docs/en/sub-agents)) | Codex explicitly prefers `model_instructions_file` or `AGENTS.md` over generic inline instructions. ([Codex config reference](https://developers.openai.com/codex/config-reference)) | Gemini uses settings files plus agent overrides. ([Gemini configuration](https://raw.githubusercontent.com/google-gemini/gemini-cli/main/docs/reference/configuration.md)) | Keep the survey rubric and artifact contract in repo-local markdown that every platform can reference. |
| Tool boundaries | Claude subagents and hooks can restrict tools. ([Claude sub-agents](https://code.claude.com/docs/en/sub-agents), [Claude hooks](https://code.claude.com/docs/en/hooks)) | Codex relies on sandbox/approval controls and MCP/tool config. ([Codex config reference](https://developers.openai.com/codex/config-reference)) | Gemini subagents and hooks can narrow or rewrite tool access. ([Gemini subagents](https://raw.githubusercontent.com/google-gemini/gemini-cli/main/docs/core/subagents.md), [Gemini hooks](https://raw.githubusercontent.com/google-gemini/gemini-cli/main/docs/hooks/reference.md)) | Write `survey` so it only needs the minimal common toolset and clear artifact-writing rules. |

## Hooks
| Lifecycle | Claude | Codex | Gemini | Common Layer |
|-----------|--------|-------|--------|--------------|
| Hook model | Claude supports shell, HTTP, and prompt hooks across many lifecycle events. ([Claude hooks](https://code.claude.com/docs/en/hooks)) | Supplied Codex docs expose a `notify` command rather than a comparable general hook framework. ([Codex config reference](https://developers.openai.com/codex/config-reference)) | Gemini supports command hooks with JSON IO and explicit exit codes across tool/agent/model/session events. ([Gemini hooks](https://raw.githubusercontent.com/google-gemini/gemini-cli/main/docs/hooks/reference.md)) | Hooks are optional wrappers, not the survey contract itself. |
| Validation fit | Claude can validate before/after tool use and around subagents. ([Claude hooks](https://code.claude.com/docs/en/hooks)) | Codex portability should assume normal post-run validation rather than hook parity. ([Codex config reference](https://developers.openai.com/codex/config-reference)) | Gemini hooks can block, rewrite, or retry around agent/tool/model steps. ([Gemini hooks](https://raw.githubusercontent.com/google-gemini/gemini-cli/main/docs/hooks/reference.md)) | Make survey validation runnable as a normal post-generation check, then wrap it in hooks where available. |

## Platform Gaps
1. Codex is the portability floor in the supplied sources: Claude and Gemini document richer subagent/hook systems, so `survey` should not depend on hook-heavy or subagent-only behavior.
2. Config formats differ materially (Claude/Gemini JSON vs Codex TOML), so the durable survey contract should live in checked-in markdown, not duplicated config blobs.
3. Hook parity is not available, so artifact validation needs a non-hook fallback.
4. Output artifacts must stay identical even when the runtime differs; platform wrappers may launch or validate the survey, but they should not change filenames, headings, or citation rules.
