# Context: changelog-maintenance structural hardening

## Workflow Context
Release-history work keeps splitting into four recurring jobs: durable `CHANGELOG.md` upkeep, audience-facing release notes, migration/deprecation updates, and lightweight game patch notes. Keep a Changelog still frames changelogs as human-readable history rather than machine dumps, while GitHub autogenerated notes, Release Drafter, and Changesets all treat automation as draft generation rather than the final editorial pass. Product-update tools add audience segmentation and visibility controls, and Steam separates major updates from smaller patch-note posts, which means one anchor still needs to decide the right artifact and route-outs instead of acting like a generic template dump.

## Affected Users
| Role | Responsibility | Skill Level |
|------|----------------|-------------|
| Repo maintainers / release engineers | Maintain durable release history and trustworthy shipped-change summaries | Intermediate–advanced |
| Product / customer comms owners | Turn shipped work into benefit-led customer updates without leaking internal noise | Intermediate |
| Docs owners / developer relations | Connect release summaries to migration, API, or help docs | Intermediate–advanced |
| Game/live-ops teams | Publish concise player-facing patch notes on the right channel | Intermediate |
| Agents using the skill catalog | Route release-writing requests to the right doc skill with low ambiguity | Mixed |

## Current Workarounds
1. Generate draft release notes from PR labels, conventional commits, or changesets, then manually rewrite the output for audience fit.
2. Keep `CHANGELOG.md` as the durable record and separately copy/edit notes into GitHub Releases, customer update hubs, or Steam patch-note posts.
3. Split breaking changes into a separate migration guide because generic release notes bury required actions.
4. Use categories, labels, exclusions, and audience segmentation rules to recover structure when automation outputs are noisy.
5. Maintain manual bullet-list patch notes for game updates even when the code release pipeline is automated.

## Adjacent Problems
- Source-of-truth drift between changelog files, GitHub Releases, docs-site announcements, and store/news posts.
- Audience collision: developers, customers, internal stakeholders, and players need different framing.
- Migration/deprecation details are frequently too large for release notes but too important to omit.
- Monorepo and multi-package workflows still need one rollup summary for stakeholders.
- Label hygiene / commit hygiene directly controls automation quality, so weak metadata leaks into public notes.

## User Voices
- “Changelogs are for humans, not machines.” — Keep a Changelog ([https://keepachangelog.com/en/1.1.0/](https://keepachangelog.com/en/1.1.0/), direct page retrieval)
- Release Drafter users still want automatic changelog sync because otherwise they “fear to miss the other” between release notes and `CHANGELOG.md` ([https://github.com/release-drafter/release-drafter/issues/866](https://github.com/release-drafter/release-drafter/issues/866), GitHub API primary-source retrieval)
- Changesets users value a root monorepo changelog because it gives stakeholders “a super quick overview of all the things that have changed in one place” ([https://github.com/changesets/changesets/issues/264](https://github.com/changesets/changesets/issues/264), GitHub API primary-source retrieval)
- LaunchNotes tells teams to explain “what changed and why it matters to your customers,” reinforcing that customer-facing notes are not raw engineering dumps ([https://help.launchnotes.com/en/articles/12902366-create-my-first-announcement-in-5-minutes](https://help.launchnotes.com/en/articles/12902366-create-my-first-announcement-in-5-minutes), direct page retrieval)
- Steamworks reminds developers that every update is a form of communication with players and still requires an explicit patch-note post ([https://partner.steamgames.com/doc/store/updates](https://partner.steamgames.com/doc/store/updates), direct page retrieval)
