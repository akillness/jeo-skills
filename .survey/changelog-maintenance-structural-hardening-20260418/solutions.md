# Solution Landscape: changelog-maintenance structural hardening

## Solution List
| Name | Approach | Strengths | Weaknesses | Notes |
|------|----------|-----------|------------|-------|
| Keep a Changelog | Human-curated changelog standard | Clear human-readable structure, explicit deprecations, durable history | Manual discipline required | Best baseline for truthful `CHANGELOG.md` upkeep |
| Changesets | Monorepo/package release intent + changelog automation | Captures intent early, strong package coordination, good for rollups | Still needs editorial cleanup and migration separation | Common in JS/TS monorepos |
| Release Drafter | Label-driven GitHub draft release notes | Good low-friction draft generation from PR metadata | GitHub-release focused; changelog sync still manual | Strong draft layer, weak final editor |
| release-please | Release PR automation from conventional commits | Reviewable release PRs, version bump + changelog automation | Sensitive to commit hygiene; manual overrides still common | Good for PR-centric release flow |
| GitHub autogenerated release notes | Built-in release-note generation | Fastest zero-install draft path | Weak for migration nuance and non-GitHub publishing surfaces | Works best with good labels/exclusions |
| semantic-release / conventional-changelog | Commit-driven release automation | Strong ecosystem adoption, automated notes and publishing | Output quality depends on commit quality | Better at collection than audience-aware editing |
| LaunchNotes / Beamer / Headway | Customer-facing release communication surfaces | Segmentation, in-app/update hubs, audience targeting | Duplicate source-of-truth risk; weak on engineering truth | Good downstream publishing layer, not canonical source |
| Steam events / patch-note posts | Game-platform update communication | Native player-facing distribution and patch-note categories | Platform-specific and manual | Distinct surface from repo changelog |

## Categories
- Human-first changelog structure
- Commit / PR / release automation
- Monorepo package-release coordination
- Migration / breaking-change communication
- Customer-facing release communication platforms
- Game patch-note publication surfaces

## What People Actually Use
Most teams mix one durable changelog convention with one automation layer and one downstream publishing surface. In practice: PR labels / commits / changesets generate drafts, then humans still regroup the final note, separate migration detail, and translate the output for customers or players.

## Frequency Ranking
1. Conventional-commit / PR automation family (`release-please`, `semantic-release`, `conventional-changelog`, Release Drafter)
2. Keep-a-Changelog-style human-curated `CHANGELOG.md`
3. Changesets for monorepos and package release coordination
4. Built-in GitHub autogenerated release notes
5. Separate migration / upgrade guides for breaking changes
6. Customer-facing update hubs (LaunchNotes / Beamer / Headway)
7. Game patch-note publication surfaces (Steam / itch.io)

## Key Gaps
- Automation solves collection better than editorial boundary selection.
- Teams still need one router that decides whether the real artifact is changelog, release notes, migration update, or lightweight patch notes.
- Product/customer tools and game surfaces are downstream channels, not source-of-truth systems.
- Existing tools do not remove the need for route-outs to internal docs, API docs, deployment execution, or launch messaging.

## Contradictions
- “Automatically generated release notes” still depend on label/config discipline and often require manual cleanup.
- Customer-facing release-note tools promise polished communication, but they do not replace repo truth or migration guidance.
- Patch-note publication feels like release-note work, but the Steam surface proves it is a distinct player-facing artifact with its own constraints.

## Key Insight
The strongest reusable move is not another changelog or release-note wrapper. It is making `changelog-maintenance` smaller and more routing-first: classify the artifact quickly, keep evidence gathering minimal but explicit, move packet structures and boundary guidance into references, and strengthen evals around route-out pressure where release writing drifts into internal docs, deployment, or launch messaging.
