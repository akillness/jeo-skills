# Creative Media Cluster

## Durable finding
The creative-media lane is healthier when one skill owns the **programmable / automated video workflow** and any narrower legacy names behave as compatibility aliases instead of pretending to be separate peers.

Current cluster shape:
- `video-production` = canonical programmable-video / automated-video anchor across Remotion, template APIs, content repurposing, QA, and publishing handoff
- `remotion-video-production` = compatibility alias for explicit Remotion naming or legacy setup surfaces

## Why it matters
- Discovery docs are high-degree nodes, so duplicate peer descriptions cause activation noise quickly.
- The real workflow boundary is not "Remotion vs video"; it is programmable / repeatable video pipelines versus manual editor-only polish.
- Support bundles matter in this lane because asset packets, QA rules, and handoff boundaries are easy to forget when the skill stays generic.

## Boundary rules
- If the question is **"how do we plan an automated or repeatable video pipeline?"** → `video-production`
- If the question is **"use the old Remotion skill name"** or the stack is explicitly Remotion and compatibility matters → `remotion-video-production`
- If the problem is only final manual polish or editor technique with no automation layer, this cluster should not pretend to own the whole job
